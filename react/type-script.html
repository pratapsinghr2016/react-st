1. Type vs Interface

Both define the shape of data, but with different capabilities.

Interface

A contract for object shapes. Can be extended and merged.

interface User {
  name: string;
  age: number;
}

// Extension
interface Employee extends User {
  role: string;
}

// Declaration merging (same name = auto-merged)
interface User {
  email: string;
}
// Now User has: name, age, email

Type

An alias for any type. More flexible - unions, primitives, tuples.

type ID = string | number;  // Union - can't do with interface

type User = {
  name: string;
  age: number;
};

// Extension via intersection
type Employee = User & { role: string };

// Tuples
type Coordinates = [number, number];
Use Interface Use Type
Object shapes Unions (string | number)
Class contracts Tuples
Declaration merging needed Primitive aliases
Public API definitions Complex type transformations

üéØ Quick Rule

Interface for objects you might extend. Type for everything else.

2. Generics

Type placeholders - "I don't know the exact type yet, but whatever type comes in, use that same type throughout."

‚ùå The Problem

function getFirstItem(arr: any[]): any {
  return arr[0];
}

const num = getFirstItem([1, 2, 3]);  // num is 'any' üòï
num.toUpperCase();  // No error, but crashes at runtime!

‚úÖ The Solution - Generics

function getFirstItem<T>(arr: T[]): T {
  return arr[0];
}

const num = getFirstItem([1, 2, 3]);  // num is 'number' ‚úÖ
const str = getFirstItem(['a', 'b']);  // str is 'string' ‚úÖ

num.toUpperCase();  // ‚ùå Error! TypeScript knows num is number

Syntax Breakdown

function myFunction<T>(param: T): T { }
//                 ‚Üë        ‚Üë     ‚Üë
//          declare it   use it  return it

Common Examples

// Identity function
function identity<T>(value: T): T {
  return value;
}

// Multiple type parameters
function swap<T, U>(pair: [T, U]): [U, T] {
  return [pair[1], pair[0]];
}

// Generic interface
interface ApiResponse<T> {
  data: T;
  success: boolean;
}

// With constraints
function logLength<T extends { length: number }>(item: T): void {
  console.log(item.length);
}

üéØ When to Use

‚Ä¢ Function works with multiple types
‚Ä¢ You want to preserve type info
‚Ä¢ Building reusable components
‚Ä¢ Relating input type to output type

3. Utility Types

Built-in helper tools that transform one type into another.

‚ùå The Problem - Repetitive Type Definitions

interface User { id: number; name: string; email: string; password: string; }

// Do I need to write these manually? üò©
interface UpdateUser { id?: number; name?: string; ... }
interface DisplayUser { id: number; name: string; email: string; }

‚úÖ The Solution - Utility Types

type UpdateUser = Partial<User>;           // All fields optional
type DisplayUser = Omit<User, 'password'>; // Everything except password

1. Partial<T> - Make All Optional

type UpdateUser = Partial<User>;
// { id?: number; name?: string; email?: string; }

// Use: Update functions
updateUser(1, { name: 'Rajat' });  // ‚úÖ Only pass what you need

2. Required<T> - Make All Required

interface Config { host?: string; port?: number; }

type FullConfig = Required<Config>;
// { host: string; port: number }

// Use: After merging with defaults

3. Readonly<T> - Can't Modify

type FrozenUser = Readonly<User>;

const user: FrozenUser = { name: 'Rajat', age: 28 };
user.name = 'John';  // ‚ùå Error! Cannot modify

4. Pick<T, Keys> - Keep Only These

type UserPreview = Pick<User, 'id' | 'name'>;
// { id: number; name: string }

// Use: When you need subset of fields

5. Omit<T, Keys> - Remove These

type SafeUser = Omit<User, 'password'>;
// Everything except password

// Use: Hide sensitive fields from API response

6. Record<Keys, Type> - Create Object Type

type Scores = Record<string, number>;
// { [key: string]: number }

const scores: Scores = { math: 95, english: 88 };

// With specific keys
type Status = 'pending' | 'approved';
type StatusColors = Record<Status, string>;
Utility What it does Example
Partial<T> All properties optional Partial<User>
Required<T> All properties required Required<Config>
Readonly<T> All properties readonly Readonly<State>
Pick<T, K> Only keep specified keys Pick<User, 'id' | 'name'>
Omit<T, K> Remove specified keys Omit<User, 'password'>
Record<K, T> Object with K keys, T values Record<string, number>

4. Type Guards (Type Narrowing)

Protection mechanism to ensure type safety at runtime. You start with a wide type and narrow it down using checks.

‚ùå The Problem

function printLength(value: string | number) {
  console.log(value.length);  // ‚ùå Error! number doesn't have .length
}

‚úÖ The Solution - Type Narrowing

function printLength(value: string | number) {
  if (typeof value === 'string') {
    console.log(value.length);  // ‚úÖ TypeScript knows it's string
  }
}

1. typeof - For Primitives

function double(value: string | number) {
  if (typeof value === 'string') {
    return value.repeat(2);    // "hi" ‚Üí "hihi"
  }
  return value * 2;            // 5 ‚Üí 10
}

2. Truthiness - For Null/Undefined

function greet(name: string | null) {
  if (name) {
    return `Hello, ${name}`;
  }
  return 'Hello, stranger';
}

3. Equality (===) - For Specific Values

function process(value: string | null) {
  if (value === null) {
    return 'No value';
  }
  return value.toUpperCase();  // value is string here
}

4. in - For Object Properties

interface Bird { fly: () => void }
interface Fish { swim: () => void }

function move(animal: Bird | Fish) {
  if ('fly' in animal) {
    animal.fly();
  } else {
    animal.swim();
  }
}

5. instanceof - For Class Instances

class Car { drive() {} }
class Bike { pedal() {} }

function operate(vehicle: Car | Bike) {
  if (vehicle instanceof Car) {
    vehicle.drive();
  } else {
    vehicle.pedal();
  }
}
Guard When to Use Example
typeof Primitives (string, number, boolean) typeof x === 'string'
Truthiness Removing null/undefined if (x) { ... }
=== Checking exact values x === null
in Objects with different properties 'bark' in animal
instanceof Class instances x instanceof Dog

5. Union & Intersection Types

Union (|) - Either This OR That

type Status = 'loading' | 'success' | 'error';

type StringOrNumber = string | number;

function print(value: string | number) {
  // value can be string OR number
}

Think: "Can be A or B"

Intersection (&) - Must Be This AND That

type Name = { name: string };
type Age = { age: number };

type Person = Name & Age;
// { name: string; age: number }

const user: Person = {
  name: 'Rajat',  // Must have
  age: 28         // Must have
};

Think: "Must be A and B combined"

Union (|) Intersection (&)
Either/or Both/and
A | B = A or B A & B = A + B merged
Flexible Strict

6. keyof & typeof

typeof - Get Type From Value

Extracts type from a JavaScript value.

const user = {
  name: 'Rajat',
  age: 28
};

type User = typeof user;
// { name: string; age: number }

Think: "What is the type of this variable?"

keyof - Get Keys From Type

Extracts keys as a union of strings.

interface User {
  name: string;
  age: number;
  email: string;
}

type UserKeys = keyof User;
// 'name' | 'age' | 'email'

Think: "What are the keys of this type?"

Combining Both

const user = { name: 'Rajat', age: 28 };

type UserKeys = keyof typeof user;
// 'name' | 'age'

// Read right to left:
// typeof user ‚Üí { name: string; age: number }
// keyof { ... } ‚Üí 'name' | 'age'

Common Use Case - Safe Property Access

function getValue<T, K extends keyof T>(obj: T, key: K): T[K] {
  return obj[key];
}

const user = { name: 'Rajat', age: 28 };

getValue(user, 'name');   // ‚úÖ returns string
getValue(user, 'email');  // ‚ùå Error! 'email' doesn't exist
Keyword Input Output
typeof Value Type
keyof Type Union of keys

7. Enums vs Const Assertions

Enum - Named Constants

// Numeric enum
enum Status {
  Pending,    // 0
  Approved,   // 1
  Rejected    // 2
}

// String enum (more common)
enum Direction {
  Up = 'UP',
  Down = 'DOWN',
  Left = 'LEFT',
  Right = 'RIGHT'
}

const move: Direction = Direction.Up;

Const Assertion (as const)

const Status = {
  Pending: 'PENDING',
  Approved: 'APPROVED',
  Rejected: 'REJECTED'
} as const;

// Get type from it
type StatusType = typeof Status[keyof typeof Status];
// 'PENDING' | 'APPROVED' | 'REJECTED'

// Array as const
const roles = ['admin', 'user', 'guest'] as const;
type Role = typeof roles[number];
// 'admin' | 'user' | 'guest'
Enum Const Assertion
TypeScript feature Plain JavaScript
Generates extra code No extra code
Built-in type Need typeof for type
Heavier Lighter (preferred)
üí° Tip: as const is preferred in modern TypeScript. Enums are still valid but add extra JavaScript code.

8. Discriminated Unions

A common property (like type or status) to tell objects apart.

type Success = { status: 'success'; data: string };
type Error = { status: 'error'; message: string };

type Response = Success | Error;

function handle(res: Response) {
  if (res.status === 'success') {
    console.log(res.data);     // ‚úÖ TypeScript knows it's Success
  } else {
    console.log(res.message);  // ‚úÖ TypeScript knows it's Error
  }
}

üéØ When to Use

API responses, Redux actions, handling different states in React components.

9. Type Assertions

"Trust me TypeScript, I know what this is."

const input = document.getElementById('name') as HTMLInputElement;

input.value = 'Rajat';  // ‚úÖ Now TypeScript knows it's an input

// Alternative syntax (not in JSX)
const input2 = <HTMLInputElement>document.getElementById('name');
‚ö†Ô∏è Warning: Use sparingly! You're telling TypeScript to trust you. If you're wrong, you'll get runtime errors.

10. unknown vs any

// any = No checking (dangerous) ‚ùå
let a: any = 5;
a.toUpperCase();  // No error, but crashes at runtime! üí•

// unknown = Must check first (safe) ‚úÖ
let b: unknown = 5;
b.toUpperCase();  // ‚ùå Error! Must check type first

if (typeof b === 'string') {
  b.toUpperCase();  // ‚úÖ Now it works
}

üéØ Rule

Always prefer unknown over any. It forces you to check the type first.
