01

Diffing Algorithm

React's diffing algorithm (Reconciliation) determines the minimum number of operations to update the DOM. It operates on O(n) complexity using two key heuristics.

01

Different Element Types: If the parent element/component type changes, React will unmount the entire subtree and mount a new one. Children are not compared ‚Äî they're destroyed and recreated.

02

Keys Identify Children: If the key of any child changes, React treats it as a completely new element and unmounts the old one. Keys help React track identity across re-renders.

üí° Why This Matters

React assumes elements of different types will produce different trees. This heuristic lets React skip expensive deep comparisons and achieve near-linear performance on tree updates.

Diffing in Action JSX 
// ‚ùå Parent type changes: entire subtree unmounts
// Before
<div><Counter /></div>
// After
<section><Counter /></section>  // Counter is unmounted & remounted!

// ‚ùå Key changes: component unmounts
<UserCard key="user-1" />
<UserCard key="user-2" />  // Complete remount, state lost

// ‚úÖ Same type, same key: React updates props, preserves state
<UserCard key="user-1" name="Alice" />
<UserCard key="user-1" name="Bob" />  // Only name prop updates

React Fiber Architecture

Fiber is React's internal reconciliation engine introduced in React 16. It reimplements the core algorithm to enable incremental rendering ‚Äî the ability to split work into chunks and spread it over multiple frames.

What is a Fiber?

A Fiber is a JavaScript object that represents a unit of work. Each React element has a corresponding Fiber node containing:

‚Üí type ‚Äî Component function/class or DOM tag name
‚Üí key ‚Äî Unique identifier among siblings
‚Üí props ‚Äî Component props
‚Üí stateNode ‚Äî DOM node or class instance
‚Üí child, sibling, return ‚Äî Links to other fibers (tree structure)

Double Buffering: Two Fiber Trees

React maintains two fiber trees for efficient updates:

Current Fiber Tree

Currently rendered UI

What the user sees on screen

DIFF

Work-in-Progress Tree

Being constructed

Next version of the UI

Simplified Fiber Node Structure JavaScript 
const fiberNode = {
  // Identity
  type: MyComponent,        // Function, Class, or 'div', 'span', etc.
  key: 'unique-key',
  
  // Data
  props: { name: 'Rajat', count: 5 },
  memoizedState: { /* hooks state */ },
  
  // Tree structure (linked list)
  child: childFiber,          // First child
  sibling: siblingFiber,      // Next sibling
  return: parentFiber,        // Parent
  
  // Work tracking
  alternate: currentFiber,    // Link to other tree version
  effectTag: 'UPDATE',       // What needs to happen (PLACEMENT, UPDATE, DELETION)
};

üîÑ Reconciliation Process

When state changes: (1) React builds a new WIP tree, (2) Compares it with the current tree (diffing), (3) Calculates necessary DOM operations, (4) Commits changes in a single batch, (5) WIP tree becomes the new current tree.

Concurrency Mechanisms

React 18 introduced concurrent features that allow React to interrupt, pause, and resume rendering work. This keeps the UI responsive during expensive updates.

useTransition()

Mark state updates as non-urgent. React keeps the UI responsive while processing in the background.

useDeferredValue()

Create a deferred version of a value that "lags behind" during heavy renders.

Suspense

Declaratively handle loading states. Components can "suspend" while waiting for data.

Automatic Batching

Multiple state updates are batched into a single re-render, even in async code.

üí° In Plain English

useTransition: "Hey React, this update isn't urgent ‚Äî do it later when you're free."

useDeferredValue: "Hey React, give me a copy of this value that can lag behind ‚Äî I don't need it to be in sync immediately."

04

useTransition()

Wraps state updates to mark them as non-urgent transitions. Urgent updates (like typing) won't be blocked by transition updates (like filtering a list).

useTransition Example ‚Äî Input + Expensive List Render JSX 
import { useState, useTransition, memo } from 'react';

function App() {
  const [text, setText] = useState('');
  const [count, setCount] = useState(0);
  const [isPending, startTransition] = useTransition();

  function handleChange(e) {
    // ‚ö° URGENT: Update input immediately (user sees their typing)
    setText(e.target.value);
    
    const len = e.target.value.length;
    
    // üê¢ NON-URGENT: List rendering can wait
    startTransition(() => {
      setCount(len);
    });
  }

  return (
    <div>
      <input type="text" onChange={handleChange} value={text} />
      {isPending && <span> Loading...</span>}
      <MemoizedList count={count} />
    </div>
  );
}

// Heavy child component ‚Äî renders (count √ó 1000) items
function List({ count }) {
  // Simulate expensive computation
  let i = 0;
  while (i < 100000000) { i++; }  // Artificial delay

  return (
    <ul>
      {Array(count * 1000)
        .fill(0)
        .map((_, idx) => <li key={idx}>Item {idx + 1}</li>)}
    </ul>
  );
}

const MemoizedList = memo(List);

üîç What's Happening Here?

Type "hello" (5 characters) ‚Üí List renders 5 √ó 1000 = 5,000 items

Without useTransition: Input freezes while List computes & renders

With useTransition: Input stays responsive, List updates in background

‚úì setText ‚Äî Urgent update, runs immediately (input stays responsive)
‚úì setCount inside startTransition ‚Äî Non-urgent, React can interrupt this
‚úì isPending ‚Äî Shows loading indicator while heavy list renders
‚úì memo(List) ‚Äî Prevents re-render when only text changes

useDeferredValue()

Creates a deferred copy of a value that "lags behind" during urgent updates. Useful when you don't control the state setter (e.g., receiving props).

useDeferredValue Example ‚Äî Same Pattern, Simpler Syntax JSX 
import { useState, useDeferredValue, memo } from 'react';

function App() {
  const [text, setText] = useState('');
  
  // üîÑ deferredText lags behind during heavy renders
  const deferredText = useDeferredValue(text);
  
  // Check if we're showing stale data
  const isStale = text !== deferredText;

  return (
    <div>
      <input
        type="text"
        value={text}
        onChange={(e) => setText(e.target.value)}
      />
      
      {/* Show what user typed ‚Äî fades when stale */}
      <p style={{ opacity: isStale ? 0.4 : 1 }}>
        Showing results for: "{deferredText}"
      </p>
      
      {/* List uses deferred value ‚Äî renders count √ó 1000 items */}
      <div style={{ opacity: isStale ? 0.4 : 1 }}>
        <MemoizedList count={deferredText.length} />
      </div>
    </div>
  );
}

// Same heavy child component from useTransition example
function List({ count }) {
  let i = 0;
  while (i < 100000000) { i++; }

  return (
    <ul>
      {Array(count * 1000)
        .fill(0)
        .map((_, idx) => <li key={idx}>Item {idx + 1}</li>)}
    </ul>
  );
}

const MemoizedList = memo(List);

üîç Visual Feedback with Opacity

User types fast: text updates instantly, deferredText lags behind

While stale: isStale = true ‚Üí UI fades to 40% opacity

When caught up: isStale = false ‚Üí UI returns to full opacity

‚úì No need to split state ‚Äî just wrap the value with useDeferredValue
‚úì text !== deferredText tells you when showing stale data
‚úì Reduced opacity provides visual feedback without a loading spinner

useTransition vs useDeferredValue

Aspect useTransition useDeferredValue
What it wraps State setter function State value
Use when You control the state update You receive a prop or can't wrap setter
Pending indicator isPending boolean Compare value !== deferredValue
Syntax startTransition(() => setState(x)) const deferred = useDeferredValue(value)

Suspense

Suspense lets components "wait" for something (typically data) before rendering, showing a fallback UI in the meantime. Works with Suspense-enabled libraries like React Query, Relay, or Next.js.

Suspense for Data Fetching JSX 
import { Suspense } from 'react';

function App() {
  return (
    <div>
      <h1>User Dashboard</h1>
      
      {/* Each Suspense boundary shows its own fallback */}
      <Suspense fallback={<ProfileSkeleton />}>
        <UserProfile />
      </Suspense>
      
      <Suspense fallback={<PostsSkeleton />}>
        <UserPosts />
      </Suspense>
    </div>
  );
}

// UserProfile uses a Suspense-enabled data source
// When data isn't ready, it "suspends" (throws a promise)
// React catches this and shows the fallback

Suspense + useTransition

When navigating between pages, combine Suspense with useTransition to keep the old UI visible while the new page loads, instead of showing a loading spinner.

Automatic Batching

React 18 automatically batches multiple state updates into a single re-render, even inside promises, timeouts, and native event handlers.

Automatic Batching in Action JSX 
function Counter() {
  const [count, setCount] = useState(0);
  const [flag, setFlag] = useState(false);

  // ‚úÖ React 18: ONE re-render (batched)
  const handleClick = () => {
    setCount(c => c + 1);  // Does not re-render yet
    setFlag(f => !f);        // Does not re-render yet
    // React batches and re-renders ONCE
  };

  // ‚úÖ React 18: ALSO batched (new in React 18!)
  const handleAsync = () => {
    fetch('/api/data').then(() => {
      setCount(c => c + 1);  // Batched together
      setFlag(f => !f);        // Single re-render
    });
  };

  // ‚úÖ React 18: Timeouts are batched too
  const handleTimeout = () => {
    setTimeout(() => {
      setCount(c => c + 1);
      setFlag(f => !f);
      // Still batched in React 18!
    }, 1000);
  };

  return <button onClick={handleClick}>{count}</button>;
}

‚ö†Ô∏è Opting Out of Batching

In rare cases where you need immediate re-renders, use flushSync from react-dom:

Force Immediate Re-render JSX 
import { flushSync } from 'react-dom';

function handleClick() {
  flushSync(() => {
    setCount(c => c + 1);
  });
  // DOM is updated here
  
  flushSync(() => {
    setFlag(f => !f);
  });
  // DOM is updated again
}

Interview Questions

Q

What is React Fiber and why was it introduced?

Fiber is React's reconciliation engine (React 16+). It's a complete rewrite of React's core algorithm that enables incremental rendering ‚Äî splitting rendering work into chunks.

Each Fiber is a JavaScript object representing a unit of work with component info (type, key, props, children) and links to parent/child/sibling fibers.

Key benefit: React can now pause, abort, or reuse work, and assign different priorities to different types of updates. This is the foundation for all concurrent features.

Q

What's the difference between useTransition and useDeferredValue?

useTransition wraps the state setter ‚Äî you use it when you control where the state is updated. It returns isPending for loading indicators.

useDeferredValue wraps the value itself ‚Äî use it when you receive data as props or can't control the setter. You detect staleness by comparing value !== deferredValue.

Both achieve the same goal: keeping urgent updates (typing) responsive while deferring expensive work.

Q

Does concurrent rendering make React faster?

Not in terms of total work ‚Äî the same computations still happen. Concurrent rendering makes the UI feel faster by ensuring high-priority updates (user input) never wait behind low-priority updates (rendering large lists).

It's about responsiveness, not raw speed. The user never experiences lag while typing, even if expensive work is happening in the background.

Q

Explain the two principles of React's diffing algorithm.

1. Different element types produce different trees: If a parent element changes type (e.g., <div> to <section>), React unmounts the entire subtree and rebuilds it. No comparison of children happens.

2. Keys identify elements across renders: When a key changes, React treats the element as completely new ‚Äî it unmounts the old one and mounts a fresh instance with fresh state.

These heuristics let React achieve O(n) complexity instead of O(n¬≥) for tree comparison.

Q

What is automatic batching in React 18?

React 18 automatically groups multiple state updates into a single re-render, even inside promises, timeouts, and native event handlers.

Before React 18, only updates inside React event handlers were batched. Now batching works everywhere, improving performance by default.

Use flushSync() from react-dom if you need to opt out and force immediate re-renders.

Q

What are the two Fiber trees and how do they work together?

React maintains two fiber trees using a technique called double buffering:

Current tree: Represents the UI currently rendered on screen.

Work-in-Progress (WIP) tree: The tree being built during reconciliation.

When a re-render completes, the WIP tree becomes the current tree, and the old current tree becomes available for reuse. The alternate property on each fiber links it to its counterpart in the other tree.
