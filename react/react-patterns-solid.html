üéØ What is SOLID?

SOLID is a set of 5 design principles that help you write maintainable, scalable, and testable code. Originally coined by Robert C. Martin (Uncle Bob) for OOP, these principles translate beautifully to React components and TypeScript.

Why it matters in interviews: Shows you think about code architecture, not just making things work.

S O L I D
S

Single Responsibility Principle

üîß One Tool, One Job
"A component should have only one reason to change."

A screwdriver screws, a hammer hammers. Don't build a Swiss Army knife component that handles auth, fetches data, and renders UI. Each component/function should do one thing well.

‚ùå Bad - Multiple responsibilities 
function UserProfile() {
  const [user, setUser] = useState(null);
  const [posts, setPosts] = useState([]);
  
  useEffect(() => {
    // Fetching user data - responsibility 1
    fetch('/api/user').then(r => r.json()).then(setUser);
    // Fetching posts - responsibility 2
    fetch('/api/posts').then(r => r.json()).then(setPosts);
  }, []);
  
  // Formatting date - responsibility 3
  const formattedDate = new Date(user?.createdAt).toLocaleDateString();
  
  // Rendering everything - responsibility 4
  return (...);
}
‚úÖ Good - Single responsibility each 
// Hook for user data only
function useUser() {
  const [user, setUser] = useState(null);
  useEffect(() => {
    fetch('/api/user').then(r => r.json()).then(setUser);
  }, []);
  return user;
}

// Utility for formatting
const formatDate = (date: string) => new Date(date).toLocaleDateString();

// Component only renders
function UserProfile() {
  const user = useUser();
  return <ProfileCard user={user} />;
}
O

Open/Closed Principle

üîå Plugin Architecture
"Open for extension, closed for modification."

Like a power strip ‚Äî you can plug in new devices without rewiring your house. Add new features through composition and props, not by editing the original component's internals.

‚ùå Bad - Modifying component for each variant 
function Button({ variant }: { variant: 'primary' | 'danger' | 'ghost' }) {
  // Every new variant = modify this component
  if (variant === 'primary') return <button className="bg-blue">...</button>;
  if (variant === 'danger') return <button className="bg-red">...</button>;
  if (variant === 'ghost') return <button className="bg-transparent">...</button>;
  // Need icon button? Edit this file again...
}
‚úÖ Good - Extend via composition 
// Base button accepts className - open for extension
function Button({ className, children, ...props }: ButtonProps) {
  return (
    <button className={`base-btn ${className}`} {...props}>
      {children}
    </button>
  );
}

// Variants extend without modifying Button
const PrimaryButton = (props) => <Button className="bg-blue" {...props} />;
const DangerButton = (props) => <Button className="bg-red" {...props} />;
const IconButton = ({ icon, ...props }) => (
  <Button {...props}>{icon} {props.children}</Button>
);
L

Liskov Substitution Principle

üîÑ Drop-in Replacement
"Subtypes must be substitutable for their base types."

Like batteries ‚Äî if something works with AA batteries, any brand of AA should work. A PrimaryButton should work anywhere a Button works, without breaking behavior.

‚ùå Bad - Subtype breaks expected behavior 
// Base input expects onChange
interface InputProps {
  value: string;
  onChange: (value: string) => void;
}

// This "specialized" input ignores onChange ‚Äî breaks substitution!
function ReadOnlyInput({ value }: InputProps) {
  // onChange is required but ignored ‚Äî caller expects it to work
  return <input value={value} readOnly />;
}
‚úÖ Good - Subtypes honor the contract 
// Clear interface with optional readonly
interface InputProps {
  value: string;
  onChange?: (value: string) => void;
  readOnly?: boolean;
}

function Input({ value, onChange, readOnly }: InputProps) {
  return (
    <input
      value={value}
      onChange={e => onChange?.(e.target.value)}
      readOnly={readOnly}
    />
  );
}

// Any Input can be used interchangeably
<Input value={text} onChange={setText} />
<Input value={text} readOnly />
I

Interface Segregation Principle

üìã Focused Menus
"Clients shouldn't depend on interfaces they don't use."

A breakfast menu shouldn't include dinner items. Don't force components to accept props they don't need. Break large interfaces into smaller, focused ones.

‚ùå Bad - Massive interface with unused props 
// Every component using this must handle ALL props
interface UserProps {
  id: string;
  name: string;
  email: string;
  avatar: string;
  role: string;
  permissions: string[];
  lastLogin: Date;
  settings: UserSettings;
  billingInfo: BillingInfo;
}

// Avatar component forced to accept billing info it doesn't need!
function UserAvatar({ name, avatar, billingInfo, ... }: UserProps) {
  return <img src={avatar} alt={name} />;
}
‚úÖ Good - Segregated interfaces 
// Small, focused interfaces
interface UserIdentity {
  id: string;
  name: string;
}

interface UserAvatar {
  name: string;
  avatar: string;
}

interface UserAuth {
  role: string;
  permissions: string[];
}

// Components only take what they need
function Avatar({ name, avatar }: UserAvatar) {
  return <img src={avatar} alt={name} />;
}

function PermissionGate({ permissions, children }: UserAuth & { children: ReactNode }) {
  return permissions.includes('admin') ? children : null;
}
D

Dependency Inversion Principle

üîå Adapters & Ports
"Depend on abstractions, not concretions."

Like how your laptop charger works in any country with the right adapter. High-level components shouldn't directly import low-level modules ‚Äî inject dependencies instead.

‚ùå Bad - Direct dependency on implementation 
import axios from 'axios'; // Tightly coupled!

function UserList() {
  const [users, setUsers] = useState([]);
  
  useEffect(() => {
    // Directly using axios ‚Äî can't easily swap or test
    axios.get('/api/users').then(res => setUsers(res.data));
  }, []);
  
  return ...;
}
‚úÖ Good - Depend on abstraction (injected) 
// Define the abstraction (interface)
interface ApiClient {
  get<T>(url: string): Promise<T>;
}

// Provide via Context
const ApiContext = createContext<ApiClient>(null!);

// Component depends on abstraction
function UserList() {
  const api = useContext(ApiContext);
  const [users, setUsers] = useState([]);
  
  useEffect(() => {
    api.get<User[]>('/api/users').then(setUsers);
  }, [api]);
  
  return ...;
}

// In app: inject real implementation
<ApiContext.Provider value={axiosClient}><App /></ApiContext.Provider>

// In tests: inject mock
<ApiContext.Provider value={mockClient}><UserList /></ApiContext.Provider>

üéÅ Bonus: Frontend-Specific Principles

  • DRY ‚Äî Don't Repeat Yourself. Extract common patterns into reusable hooks/components.
  • WET ‚Äî Write Everything Twice. Don't abstract too early ‚Äî wait for patterns to emerge.
  • KISS ‚Äî Keep It Simple, Stupid. Simple code is easier to maintain than clever code.
  • YAGNI ‚Äî You Aren't Gonna Need It. Don't build features "just in case".
  • Composition > Inheritance ‚Äî React favors composing components over extending them.
  • Colocation ‚Äî Keep related code together (component + styles + tests + types).
  • Least Privilege ‚Äî Components should only access data they actually need.
